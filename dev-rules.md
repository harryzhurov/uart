# Ведение проекта

## Термины и условные обозначения

| Название     |  Описание     |
|--------------|---------------|
| Основная ветка | Ветка, содержащаяся в репозитории на протяжении всей его жизни. К таким веткам относятся master и dev |
| Рабочая ветка  | Временная ветка, предназначенная для выполнения конкретной работы, по завершении которой изменения из рабочей ветки сливаются в основную, а сама рабочая ветка удаляется |
| master         | Стабильная ветка, содержит **только** релизы |
| dev            | Главная ветка разработки, содержит работоспособные состояния проекта |
| fb-            |  Ветка развития (feature branch). Ветки с таким префиксом являются основными, в которых ведётся разработка  новых возможностей  |
| fix-           | Ветка правок (hotfix branch). Ветки для исправления ошибок и других срочных правок  |
| Комит слияния (КС) | Комит (фиксация), возникающий при слиянии двух (или более) веток при выключенной "перемотке" (fast-forward). КС имеет два (или более) предка и явно обозначает точку слияния веток, что делает историю хранилища значительно более строгой и понятной|

## Главные ветки

Репозиторий содержит две постоянные ветки:

* `master`
* `dev`

Ветка `master` предназначена только для публикации релизов, т.е. оттестированных целостных состояний проекта. Ведение разработки - т.е. создания "черновых" рабочих фиксаций в ветке `master` не допускается.

Ветка `dev` также не предназначена для ведения непосредственно в ней разработки - она, так же как и `master`, содержит только целостные состояния, каждое из которых как минимум может быть собрано без ошибок. Как правило, фиксации ветки `dev` представляют собой "комиты слияния" (см ниже) из других веток, которые добавляют новые возможности и/или вносят какие-либо правки.

Основное отличие ветки `dev` от ветки `master` состоит в том, что уровень стабильности ("оттестированности") в ветке `dev` ниже, её состояние соответствует beta версии проекта. Фиксации ветки `dev`, которые признаны достаточно стабильными для релиза, сливаются в ветку `master`. При этом на ветке `master` ставится соответствующая метка, обозначающая версию релиза.

Такой подход даёт возможность свободно работать с проектом во рабочих ветках (см ниже), не опасаясь что-либо "сломать", при этом всегда имея под рукой и стабильный релиз `master`, и ветку для оперативной разработки `dev`.

## Рабочие ветки

Рабочие ветки являются относительно короткоживущими и предназначены для проведения в них всех основных изменений частей проекта. Как правило, это ветки, в которых разрабатывается что-то новое (feature branches) и ветки, в которых ведутся правки (hotfix branches).

Типичная схема работы с ветками такова (предполагается, что текущее, более-менее стабильное, состояние проекта зафиксировано в ветке `dev`): при возникновении необходимости или желания что-либо делать с кодом (разрабатывать новую функциональность, править ошибку, поэкспериментировать и т.д.) создаётся новая ветка, в ней спокойно ведётся вся работа, по завершении работы результат сливается обратно в ветку, от которой было сделано это временное ответвление, после чего временная ветка удаляется.

Таким образом, эта схема позволяет в любой момент делать что угодно с кодом, не опасаясь что-либо потерять, и иметь возможность в любой момент практически мгновенно переключиться на любую стабильную фиксацию. Второе достоинство состоит в том, что выстраивается достаточно красивая история разработки: на графе лога отчётливо видны этапы в виде цепочек фиксаций, слитых в постоянную ветку.

Основные типы рабочих веток:

* Ветка развития (Feature Branch)
* Ветка правок (Fix Branch)

Рекомендуется при создании ветки давать её имя, состоящее из префикса, отражающего функциональную направленность ветки. Например, для веток развития (feature branches) префиксом может быть fb-, а для правок - fix-. Поскольку рабочие ветки часто не покидают локальных репозиториев разрабочтиков, а если и покидают (т.е. публикуются на удалённом публичном сервере), то "живут" там относительно недолго, каких-то жёстких правил в отношении их именования нет, а указанные префиксы - это совет-рекомендация, и разработчик сам решает, как ему их именовать.

Технически оба типа веток создаются и завершаются одинаково [типовой вариант] - от ветки `dev`.

### Пример использования рабочей ветки

Пусть текущее состояние рабочей директории соответствует фиксации в ветке `dev`. Требуется добавить какую-то новую функциональность в проект. Для этого первым шагом создаётся новая ветка на основе текущей ветки (фиксации):

```
$ git checkout dev           // убедиться, что находимся на ветке разработки dev
$ git checkout -b fb-xxx         // создать новую ветку с именем fb-xxx
```

Если ветка используется для совместной работы либо по какой-то причине хочется сделать её публичной, можно отправить её на удалённый сервер:

```
$ git push -u origin fb-xxx      // опубликовать ветку на удалённом сервере
```

Далее производится обычная работа над кодом, фиксируются все необходимые изменения, и когда работа завершена, т.е. новая функциональность готова к тому, чтобы быть включенной в основную ветку разработки, делается слияние:

```
$ git checkout dev           // переключение на ветку dev
$ git merge --no-ff fb-xxx       // слияние, опция --no-ff нужна, чтобы возник комит слияния
$ git push origin dev        // публикация результата
```

Если не указать опцию --no-ff (no fast-forward), то git сделает слияние по схеме "перемотка" (fast-forward) - как бы встроит фиксации из рабочей ветки в dev, а это не то, что нужно. Для более чёткой и красивой истории хочется видеть этапность на графе лога, поэтому слияние нужно проводить так, чтобы образовался так называемый комит слияния - отдельная фиксация, имеющая двух предков (из каждой сливаемой ветки).

Дополнительно команде `merge` можно указать опцию `-e`, которая вызовет текстовый редактор для написания комментария, без этой опции слияние пройдёт с автоматическим комментарием "Merge branch xxx".

При слиянии изменений в ветку `master` всё делается ровно по этой же схеме за исключением того, что ветка, в которую делается слияние - это `master`, а ветка, изменения из которой сливаются, - это `dev`. 

# Комментарий при создании фиксации

Комментарий при создании фиксации должен иметь следующий формат:
```
add feature xxx to yyy

Commit message body, prepended with blank line.
Some more descriptive message for commit, 
Max line length is ~80 symbols.
 * bulleted lists allowed;
 - bullets can be different chars

Message can contain more than one paragraph.
```

Первая строка комментария:

* должна быть коротким описанием желательно не длиннее 50 символов;
* может начинаться со строчной (предпочтительно) или заглавной буквы;
* должна описывать фиксируемые изменения в императивном стиле, например "make xyzzy do frotz" вместо "[This patch] makes xyzzy do frotz";

Если комментарий должен быть более подробным, чем это может быть выражено одной строкой, то ниже, обязательно через пустую строку, можно написать подробный текст комментария. Текст пишется в произвольной форме, длину строк желательно делать не более 80 символов, допускаются списки, в т.ч. маркированные (начинающиеся с символов "*", "-") или нумерованные. 

**ЗАМЕЧАНИЕ**

Комиты за редким исключением должны быть атомарными - т.е. каждый комит должен включать правки к одному законченному по смыслу изменению. Если при составлении комита комментарий прерващается в сложное предложение (с союзами "и" или перечислением изменений через запятую), то это является явным признаком того, что комит является неатомарным, и фиксируемые правки надо разбивать на отдельные комиты.

Пустая строка между первой строкой комментария и его расширенной частью (при наличии оной) нужна для аккуратного отображения комментариев при выводе лога в режиме комментариев в одну строку, например:

```
git log --oneline
```
или используя другие опции:

```
git log --pretty=format...
git log --decorate --format=format:...
```
